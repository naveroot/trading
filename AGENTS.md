See [PROJECT.md](PROJECT.md) for product scope: **AtlasQuant** — сбор рыночных данных по инструментам (котировки, объёмы, ликвидность, фандинг и др.) и воспроизводимая отчётность по правилам пользователя (период, фильтры, сортировка, агрегации).

## Stack
Ruby on Rails 8, PostgreSQL, RSpec, Docker

## Key commands
- `bin/setup` — bootstrap
- `bin/rails s` — run server
- `bundle exec rspec` — run tests
- `bin/rails db:migrate` — migrate

## Architecture (consensus)
Цель — предсказуемые границы ответственности и тестируемость без «свалки» логики в контроллерах и без отдельного слоя **service objects**.

### Слоистая модель
- **Интерфейс (delivery):** контроллеры, views, serializers/helpers — только HTTP, маршрутизация, форматы ответа; без бизнес-правил.
- **Прикладной слой (application):** сценарии использования — оркестрация шагов (например «собрать котировки за период», «собрать отчёт») через явные входы/выходы. Предпочтительно: **Form objects / Command-like POROs** под конкретный кейс, **ActiveJob** для фона и повторяемости, **Query objects** для сложной выборки — всё с понятными именами и одной ответственностью.
- **Домен:** инварианты, правила отбора/агрегации, значимые для AtlasQuant сущности и расчёты. Держать в **явных доменных пространствах имён** (например `app/models/concerns/...`, `app/models/<domain>/`, или отдельный корень `app/domain/<bounded_context>/` — выбрать один стиль в проекте и придерживаться его). ActiveRecord-модели — граница персистентности; тяжёлую предметную логику выносить в модули/классы домена, вызываемые из моделей или из прикладного слоя.
- **Инфраструктура:** клиенты внешних API, адаптеры к поставщикам данных, ретраи/таймауты — отдельно от домена; домен получает данные через интерфейсы (простые Ruby-модули/классы), реализации живут рядом с интеграциями.

### Явные домены (bounded contexts)
Разделять как минимум области вокруг **источников данных / ingestion**, **инструментов и метаданных**, **отчётов и представления**, **расписания и фоновых задач** — имена папок и модулей должны отражать эту грань, чтобы не смешивать правила отчёта с деталями конкретного брокерского API.

### Чего избегать
- **Service objects** как универсальный слой «вызови `SomethingService.call`» — не вводить; вместо этого — доменные типы, команды под кейс, джобы, запросы.
- Толстые контроллеры и скрытая бизнес-логика во view.

### Перекрёстные принципы (кратко)
- **Производительность:** индексы и осознанные запросы; тяжёлое — в фоне; избегать N+1 в отчётах.
- **Безопасность:** секреты только через Rails credentials / ENV; не логировать ключи и полные ответы провайдеров с персональными данными.
- **UX отчётов:** предсказуемые поля, явные единицы и периоды, понятные ошибки при сбое сбора данных.
- **Тесты:** RSpec на доменные правила и границы слоёв; фикстуры — FactoryBot; критичные сценарии ingestion/отчёта — интеграционно там, где есть реальная ценность.

## Conventions
- Rails MVC как каркас; бизнес-логика — в домене и прикладном слое по правилам выше, не в «сервисах».
- RSpec for tests, FactoryBot for fixtures
- No new gems without explicit request

## Constraints
- Don't touch existing migrations
